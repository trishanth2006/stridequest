Notification Engine — Architecture Approaches

Which library strategy fits best? Review trade-offs, then pick your preferred direction.

A

Notifee + expo-notifications (Hybrid) ★ Recommended

@notifee/react-native owns all local notification display (foreground service, channels, styled notifications). expo-notifications handles push token registration & FCM token management. Supabase Edge Function delivers background "Lost Territory" pushes via FCM → received by expo-notifications → displayed by Notifee.

Pros

  • Notifee's Android Foreground Service is best-in-class
  • expo-notifications integrates cleanly with EAS for push token lifecycle
  • Covers all 3 blocks: live run + smart events + background push
  • Well-documented combo in Expo community

Cons

  • Two libraries to learn & maintain
  • Slight coordination overhead between the two
B

Notifee Only (Local-first)

@notifee/react-native handles everything — foreground service, local event notifications. Background "Lost Territory" is delivered via Supabase Realtime (WebSocket) when the app is alive. No FCM/push pipeline needed.

Pros

  • Single library, simpler setup
  • No server-side push infrastructure
  • Realtime subscription covers most use cases

Cons

  • App must be open/foreground for "Lost Territory" to arrive
  • Doesn't satisfy your Block 2 requirement B (background push)
  • Realtime WebSocket drains battery if always-on
C

Notifee + React Native Firebase

@notifee/react-native + @react-native-firebase/messaging for FCM directly. This is the pattern used by large-scale production apps. More powerful, more control.

Pros

  • Maximum control over FCM lifecycle
  • Battle-tested in high-scale apps
  • Background message handlers are more reliable

Cons

  • Adds Firebase SDK (~4MB) + google-services.json setup
  • Significantly more complex EAS config plugin wiring
  • Overkill for MVP scope
  • Harder to iterate on

My recommendation

Option A. Notifee gives the Android Foreground Service we need; expo-notifications is already the idiomatic Expo push token layer and avoids pulling in the full Firebase SDK. The two libraries have well-defined, non-overlapping responsibilities.
