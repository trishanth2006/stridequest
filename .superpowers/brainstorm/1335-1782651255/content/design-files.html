LiveRunNotification.ts

  • Create/configure Android notification channel
  • Start foreground service via Notifee
  • Update notification content (throttled 5s)
  • Flip to "Paused" UI state
  • Stop service + show summary notification

EventNotificationQueue.ts

  • Accept territory / quest / XP events
  • Buffer territory captures (60s window)
  • Flush as single batched notification
  • Guard against duplicate quest notifications

NotificationManager.ts (façade)

  • Single import for record.tsx and layout
  • Re-exports start / update / pause / resume / stop
  • Re-exports enqueue* event methods
  • < 50 lines — pure delegation

usePushRegistration.ts

  • Request notification permission (iOS + Android 13+)
  • Get FCM token via expo-notifications
  • Upsert token into push_tokens table
  • Called once at app startup in _layout.tsx

Why a notifications feature folder (not inside running/)?

The notification system spans multiple features: running (live run), territory (captures/losses), quests (completion). Owning it inside running/ would create the wrong dependency direction. A dedicated notifications/ feature is the correct home.
